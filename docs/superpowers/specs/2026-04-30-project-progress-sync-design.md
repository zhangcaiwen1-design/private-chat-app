# 项目总控台双轨进度同步系统设计方案

日期：2026-04-30

## 1. 设计目标

本方案为“项目组合总控台”增加进度同步能力，让总控台不仅展示静态项目卡片，还能根据用户电脑上的项目开发状态和用户亲自确认的现实事项，持续更新每个项目的进展、阻塞和下一步。

核心目标：

> 自动采集电脑里的开发进度，人工记录现实世界办理进展，合并展示到项目总控台。

该系统服务于当前五个项目：

1. 冻品开单助手
2. AI 女友
3. 私密聊天助手
4. 金点子
5. 天缘

## 2. 核心原则

### 2.1 自动采集和人工确认必须分开

项目进展分为两类：

1. **自动采集进展**
   来自本地代码仓库、文件、文档、提交记录、分支状态。

2. **人工确认进展**
   来自用户亲自办理或外部平台发生的事项，例如软著、备案、上架、店铺、推广、成交。

系统不能把自动采集结果误判成现实事项已完成。

例如：

- 能自动判断：某个项目最近有代码改动。
- 不能自动判断：应用商店审核已经通过。

### 2.2 自动轨用于开发判断，人工轨用于现实确认

自动轨回答：

- 哪个项目最近在开发？
- 哪个项目未提交改动很多？
- 哪个项目有计划文档但没有实现？
- 哪个项目可能需要测试、整理、提交？

人工轨回答：

- 公司变更是否完成？
- 软著是否提交/通过？
- ICP/App 备案是否通过？
- 店铺是否开通？
- 达人是否联系？
- 是否有咨询、成交、退款？

### 2.3 第一版用手动触发，不做后台常驻

第一版不做后台定时扫描，不做系统级监控，不做自动发帖或自动联系外部人员。

第一版更新方式：

- 用户说：“更新项目总控台。”
- Claude 扫描 `D:/coding` 下项目。
- Claude 更新自动快照。
- 用户说现实进展。
- Claude 更新人工进展。

这样安全、可控、可解释。

## 4. 自动采集轨

### 4.1 数据来源

自动采集从本地项目目录读取：

- Git 信息
- 文件更新时间
- 文档目录
- specs/plans 文件
- package.json / Gradle / UniApp 等项目特征
- 测试文件
- 未提交改动

### 4.2 采集字段

每个项目生成一条自动快照：

```json
{
  "projectId": "private-chat",
  "name": "私密聊天助手",
  "path": "D:/coding/private-calculator-chat",
  "isGitRepo": true,
  "branch": "task-5-persist-chats",
  "lastCommit": {
    "hash": "673cdf0",
    "message": "docs: specify soft premium chat ui",
    "date": "2026-04-28"
  },
  "workingTree": {
    "modified": 4,
    "added": 0,
    "deleted": 0,
    "untracked": 10
  },
  "docs": {
    "hasSpecs": true,
    "hasPlans": true,
    "recentSpecCount": 3,
    "recentPlanCount": 2
  },
  "tech": {
    "type": "Expo/React Native",
    "hasPackageJson": true,
    "hasGradle": false,
    "hasTests": false
  },
  "inferredStatus": "开发与规划并行",
  "suggestedNextAction": "整理当前未提交改动，确认哪些属于本次总控台/上架工作流。",
  "updatedAt": "2026-04-30"
}
```

实际字段可以按项目情况生成，但语义必须稳定。

### 4.3 自动推断状态

根据扫描结果推断：

| 条件 | 推断状态 |
|---|---|
| 最近有 specs/plans，代码改动少 | 文档规划中 |
| 最近代码改动多，未提交多 | 开发中，需整理提交 |
| 有计划文档，有对应代码改动 | 计划执行中 |
| 长期无提交、无文件变化 | 暂停/低活跃 |
| 有测试文件但未运行 | 需要验证 |
| 多个工作树/分支活跃 | 多线并行，需合并视图 |

自动推断只作为建议，不直接覆盖人工确认状态。

### 4.4 自动轨输出文件

输出到：

`docs/project-progress-snapshot.json`

该文件由扫描生成，供总控台读取或供 Claude 总结。

## 5. 人工确认轨

### 5.1 数据来源

人工确认轨来自用户自然语言，例如：

- “我已经把域名买了，网址是 yifan1.com。”
- “这个域名是个人名义买的。”
- “我预约了 5 月 6 号让代账会计帮我变更营业执照名称和经营范围。”
- “软著已经提交。”
- “备案被退回，原因是主体不一致。”
- “淘宝店开好了。”
- “今天有人咨询。”

### 5.2 人工事项字段

人工进展记录格式：

```json
{
  "id": "manual-20260430-company-change",
  "projectId": "private-chat",
  "module": "上架与合规",
  "item": "公司名称变更和经营范围增项",
  "status": "已预约",
  "date": "2026-04-30",
  "scheduledDate": "2026-05-06",
  "owner": "用户",
  "requiresUserAction": true,
  "isBlocking": true,
  "note": "用户已预约 2026-05-06 让代账会计办理公司名称变更和经营范围增项。",
  "nextAction": "2026-05-06 后确认办理结果，并更新新营业执照信息。"
}
```

### 5.3 当前已知人工事项

当前应记录：

1. `yifan1.com` 已由个人名义购买。
   注意：不能直接视为公司主体备案完成。后续若用于公司 App 上架、备案、协议页，需要确认是否转公司实名/备案，或另购公司域名。

2. 公司名称变更和经营范围增项已预约。
   用户已预约 2026-05-06 让代账会计办理公司名称变更和经营范围增项。

### 5.4 人工轨输出文件

输出到：

`docs/manual-progress-log.json`

该文件作为现实世界进展记录，不应由自动扫描覆盖。

## 6. 合并展示层

### 6.1 总控台项目卡片新增内容

每个项目卡片显示两类进度：

#### 自动开发进度

- 最近活跃时间
- 当前分支
- 未提交改动数量
- 最近提交
- 推断开发状态
- 自动建议下一步

#### 人工现实进度

- 外部事项状态
- 是否需要用户本人操作
- 是否阻塞
- 预约/截止日期
- 下一步

### 6.2 标记方式

建议使用：

- `AUTO`：自动采集
- `MANUAL`：人工确认
- `USER`：需要用户本人操作
- `BLOCKED`：阻塞项
- `READY`：可以交给 Claude 处理

### 6.3 下一步建议合成规则

总控台下一步建议按优先级排序：

1. P0 项目的阻塞事项。
2. 必须用户本人操作的事项。
3. 自动轨发现的高风险事项，例如大量未提交改动。
4. 可以交给 Claude 处理的文档、协议、代码整理、推广素材。
5. 低优先级项目的痛点雷达补全。

## 7. 更新交互方式

### 7.1 自动更新触发

用户说：

> 更新项目总控台

Claude 执行：

1. 扫描 `D:/coding` 下项目。
2. 更新 `docs/project-progress-snapshot.json`。
3. 汇总变化。
4. 更新 `docs/master-project-dashboard.html` 的展示数据。
5. 告诉用户哪些项目有变化。

### 7.2 人工事项更新

用户说：

> 软著提交了，名称是 XXX。

Claude 执行：

1. 判断所属项目。
2. 写入 `docs/manual-progress-log.json`。
3. 更新相关子工作流。
4. 必要时保存长期记忆。
5. 更新总控台显示。

### 7.3 启动提醒

可通过 Claude Code SessionStart hook 提醒：

- 总控台地址。
- 当前 P0 项目。
- 需要用户本人操作的事项。
- 是否要执行一次“更新项目总控台”。

提醒只提示，不自动扫描，除非用户要求。

## 8. 权限和安全边界

### 8.1 自动扫描限制

自动扫描只读取：

- Git 状态
- 文件路径和更新时间
- 项目配置文件
- 文档数量和路径

自动扫描不读取：

- `.env`
- 密钥文件
- 证书文件
- 大型二进制文件
- 用户私人非项目目录

### 8.2 外部动作限制

第一版不做：

- 自动提交代码
- 自动推送仓库
- 自动联系达人
- 自动发帖
- 自动开店
- 自动提交软著/备案/应用商店
- 自动操作支付平台

这些必须由用户确认或亲自操作。
