# 项目组合总控台 v4 设计方案

日期：2026-05-01

## 1. 设计目标

项目组合总控台 v4 的目标，是把“爆款 APP 生产工作流”从单项目流程升级为多项目并行的创业操作系统。

用户不是只做一个 App，而是在同时推进多个处于不同生命周期的项目：

1. 冻品开单助手：开发已较完整，下一步重点是推广、上架、真实试用和收费验证。
2. 私密聊天助手：已有产品雏形，下一步重点是 MVP 打磨、上架合规、商业闭环。
3. 金点子：把“爆款 APP 生产工作流”产品化，未来可对外服务其他 App 创业者。
4. AI 女友：处于痛点雷达和机会评分阶段。
5. 天缘：处于痛点雷达和机会评分阶段。

v4 不要求所有项目每天做同一种事，而是让每个项目按照当前阶段进入对应轨道，并在每天打开总控台时给出最该推进的动作。

核心目标：

> 每天打开总控台，就知道哪个项目最接近赚钱，哪个事项最卡住，哪些事可以交给 Claude，哪些项目只需保持观察。

## 2. 核心原则

### 2.1 项目组合优先，不做单线排队

多个项目可以同时推进，因为它们处在不同阶段。冻品开单助手可以进入推广验证，私密聊天助手可以推进 MVP 和合规，AI 女友和天缘可以继续做机会验证，金点子可以沉淀方法论。

系统不应该强迫所有项目按同一节奏前进。

### 2.2 按生命周期分轨，而不是按项目名称分组

项目的下一步由阶段决定，而不是由项目本身决定。

同一个项目可以在不同时间进入不同轨道。例如私密聊天助手现在在 MVP 轨和上架合规轨，未来会进入推广验证轨。冻品开单助手现在更靠近推广验证轨和上架合规轨。

### 2.3 优先级采用混合排序

总控台不只看“重要性”，而是同时看：

1. 谁离赚钱最近。
2. 谁有用户本人必须处理的阻塞。
3. 谁卡住后会影响后续一串动作。
4. 哪些事项可以立刻交给 Claude 推进。
5. 哪些项目需要保持活跃但不应占用主注意力。

## 3. 五条项目轨道

### 3.1 机会轨

适合项目：AI 女友、天缘。

作用：判断项目值不值得进入 MVP。

输入：用户经验、真实痛点、目标人群、竞品观察、渠道线索。

输出：

- 痛点雷达卡。
- 10 维项目评分。
- 目标人群定义。
- MVP 候选方案。
- 合规风险判断。
- 下一步验证动作。

### 3.2 MVP 轨

适合项目：私密聊天助手。

作用：把想法做成可以演示、可以试用、可以收费验证的最小版本。

输出：

- 核心功能闭环。
- 15 秒演示路径。
- 会员价值点。
- 付费入口。
- 赠送/邀请机制。
- 当前版本可测试清单。

私密聊天助手当前 MVP 判断：计算器入口、微信式聊天、一键锁回、9.9 元/月、赠送 TA 月卡，是当前最重要的商业演示闭环。

### 3.3 上架合规轨

适合项目：私密聊天助手、冻品开单助手。

作用：把项目从“本地能跑”推进到“能提交平台审核”。

输出：

- 公司主体状态。
- 经营范围状态。
- 域名和备案状态。
- 软著名称与材料。
- 隐私政策、用户协议、会员协议。
- 权限清单、SDK 清单。
- 商店文案、截图、测试账号。
- 审核反馈处理记录。

上架合规轨必须区分自动开发进度和人工现实进度。代码完成不等于软著完成，文档生成不等于备案通过。

### 3.4 推广验证轨

适合项目：冻品开单助手优先，私密聊天助手随后进入。

作用：让真实用户看到、试用、咨询、付费或反馈。

输出：

- 目标渠道清单。
- 私域/微信群试用方案。
- 咸鱼、淘宝、拼多多、抖音素材。
- 商品标题、主图卖点、短视频脚本。
- 询单话术。
- 试用反馈表。
- 成交、咨询、退款、留存记录。

冻品开单助手当前判断：开发较完整，应优先进入真实试用和推广验证，不应继续只在代码层打磨。

### 3.5 产品化轨

适合项目：金点子。

作用：把内部跑通的爆款 APP 生产流程沉淀成可以复用、可以展示、未来可以对外销售的产品。

输出：

- 标准模板。
- 项目评分器。
- 痛点雷达表。
- 阶段工作流。
- 总控台看板。
- 案例库。
- 对外产品定位。

金点子不应脱离真实项目空转。它应从冻品开单助手、私密聊天助手、AI 女友、天缘的真实推进过程里长出来。

## 4. 每日总控台四块内容

每天打开总控台，优先显示四块内容。

### 4.1 今天最该赚钱的项目

定义：离推广、试用、成交、收费最近的项目。

当前建议：冻品开单助手。

原因：冻品开单助手开发已较完整，下一步是找真实冻品批发用户试用、推广、上架和收费验证。

### 4.2 今天最卡住的事项

定义：不处理就会挡住后续一串动作的事项，尤其是必须用户本人处理的现实事项。

当前建议：私密聊天助手上架合规事项。

原因：公司名称、经营范围、域名主体、软著名称、新营业执照、备案材料，会影响后续上架和商业化。

### 4.3 今天可以交给 Claude 的事项

定义：不需要用户本人办理，Claude 可以直接推进的文档、整理、分析、素材和代码任务。

当前可做：

- 私密聊天助手：协议草稿、软著说明、商店文案、权限说明、SDK 清单、截图脚本。
- 冻品开单助手：推广话术、试用流程、同行微信群邀请文案、上架材料、客户反馈表。
- 金点子：把已跑过的流程沉淀成模板。

### 4.4 今天保持观察的项目

定义：需要保持活跃但不应抢占 P0 注意力的项目。

当前建议：AI 女友、天缘、金点子。

- AI 女友：继续补痛点雷达、合规边界、MVP 候选。
- 天缘：继续补本地交友征婚的目标用户、真实性机制、线下获客和合规风险。
- 金点子：从真实项目中沉淀，不急着独立产品化。

## 5. 优先级算法

### 5.1 P0

满足任意条件即可进入 P0：

1. 离收费、推广、真实试用最近。
2. 有用户本人必须处理的现实阻塞。
3. 有外部窗口期，例如预约、审核、客户咨询、试用机会。
4. 如果不处理，会拖住后续多个阶段。

当前 P0 候选：

- 冻品开单助手推广验证。
- 私密聊天助手上架合规阻塞。

### 5.2 P1

Claude 可以直接推进，且能为 P0 项目清路的事项。

示例：

- 写协议。
- 整理软著材料。
- 生成推广文案。
- 做商店介绍。
- 整理试用话术。
- 做竞品分析。
- 更新总控台。

### 5.3 P2

保持活跃但不抢主线注意力的事项。

示例：

- 补痛点雷达。
- 明确目标人群。
- 设计 MVP 候选。
- 降低合规风险。
- 找推广渠道假设。

## 6. 当前项目分配

| 项目 | 当前轨道 | 当前优先级 | 下一步 |
|---|---|---|---|
| 冻品开单助手 | 推广验证轨 + 上架合规轨 | P0 | 设计试用推广方案，准备首批真实用户触达 |
| 私密聊天助手 | MVP 轨 + 上架合规轨 | P0 | 继续 MVP 打磨，推进公司主体、软著、域名、协议 |
| 金点子 | 产品化轨 | P1/P2 | 从真实项目沉淀模板和总控台，不急于独立开发 |
| AI 女友 | 机会轨 | P2 | 补痛点雷达、目标人群、合规边界 |
| 天缘 | 机会轨 | P2 | 补本地用户画像、真实性机制、线下获客路径 |

## 7. 总控台 v4 成功标准

v4 完成后，总控台应该能回答：

1. 今天哪个项目最接近赚钱。
2. 今天哪个事项最卡住。
3. 哪些事项必须用户本人处理。
4. 哪些事项可以交给 Claude 做。
5. 每个项目当前在哪条轨道。
6. 每个项目下一步最小动作是什么。
7. 金点子如何从真实项目里沉淀，而不是空转。

## 8. 不做什么

v4 不做：

- 自动上传 GitHub。
- 自动部署。
- 自动提交应用商店。
- 自动联系客户或达人。
- 自动判断现实事项已经完成。
- 把所有项目强行排成同一个线性流程。

所有现实世界动作仍然需要用户确认或亲自执行。

## 9. 当前推荐下一步

先把 v4 总控台做成内部可用版本：

1. 在总控台数据里给每个项目增加当前轨道、优先级、下一步最小动作。
2. 在总控台首页增加每日四块内容：最该赚钱、最卡住、可交给 Claude、保持观察。
3. 让冻品开单助手进入推广验证轨。
4. 让私密聊天助手继续在 MVP 轨和上架合规轨推进。
5. 让金点子只沉淀模板，不抢 P0 资源。

一句话：

> 多项目并行不是平均用力，而是让每个项目在自己的阶段上每天前进一步。
