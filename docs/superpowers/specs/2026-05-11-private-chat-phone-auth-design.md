# Private Chat 手机号账号体系设计

## 目标

把当前以本地匿名身份为主的私密聊天 App，改为以手机号账号体系为主：

- 首次在设备上完成手机号注册/登录
- 后续保持长期登录态，不再重复输入手机号登录
- 进入链路仍保留计算器伪装页与本地进入密码
- 好友体系从用户 ID / 二维码切换为手机号搜索添加
- 聊天数据默认上云并归属于账号

## 范围

本次设计覆盖：

- 手机号 + 密码注册登录
- 本地进入密码后的登录态分流
- 注册后补资料
- 本地旧数据绑定或清空选择
- 手机号添加好友
- 单设备登录与挤下线
- 聊天数据从本地匿名归属迁移到账号归属
- 会员页、状态栏、旧好友入口等关联 UI 调整

本次不覆盖：

- 真实短信验证码登录
- 自助短信找回密码
- 会员权益细化
- 二维码好友体系重做

## 入口与页面流转

### 总体入口

1. 用户打开 App，先进入计算器页。
2. 用户输入本地进入密码。
3. 进入密码通过后按本机登录态分流：
   - 本机已有有效登录态：直接进入主界面。
   - 本机没有有效登录态：进入手机号体系。

### 手机号体系

采用“先输手机号，再由后端判断登录或注册”的单入口方式：

1. 输入手机号。
2. 后端判断该手机号是否已存在。
3. 若已存在，进入密码登录页。
4. 若不存在，进入注册流程。

### 注册流程

1. 输入手机号。
2. 设置登录密码并完成注册。
3. 进入补资料页：
   - 昵称必填。
   - 头像支持相册选择和拍照。
   - 头像可跳过。
4. 若本地已有旧聊天数据，进入本地数据处理选择页：
   - 绑定当前本地聊天数据到当前账号。
   - 清空本地数据后开始。
5. 完成后进入主界面。

### 登录成功后的行为

- 登录成功后，本机保存长期登录态。
- 后续再次打开 App，不需要重新输入手机号登录。
- 只有换设备时，才重新走手机号登录/注册流程。
- 若用户主动退出登录，则清除登录态并在下次进入时重新走手机号体系。

## 账号模型

### 对外账号

- 对外使用手机号作为唯一账号入口。
- 登录方式为手机号 + 密码。
- 找回密码先采用“联系开发者找回”的人工方式。

### 对内标识

- 内部仍保留 `userId` 作为系统主键。
- `userId` 只用于数据库关系、聊天归属、好友关系等内部用途。
- 前端不再向用户暴露 userId，也不再用 userId 作为加好友方式。

## 好友体系

### 替换原则

移除以下旧入口：

- 用户 ID 添加好友
- 我的二维码
- 扫一扫二维码加好友

仅保留手机号搜索添加好友。

### 手机号添加流程

1. 用户输入对方手机号。
2. 后端搜索是否存在对应账号。
3. 如果找到：
   - 返回对方昵称与头像。
   - 不返回可供前端展示的手机号。
   - 用户确认后再发送好友申请。
4. 如果没找到：
   - 统一提示“未找到用户”。
   - 不暴露该手机号是否已注册。

## 聊天数据与存储规则

### 归属规则

- 登录后，聊天数据归属于当前账号。
- 所有用户聊天默认上云。
- 本地仍保留缓存，但缓存属于当前账号，不再属于匿名本地身份。

### 首次接入账号时的旧数据处理

当当前设备首次完成注册或登录，且本地已有旧匿名聊天数据时，必须弹出选择：

- 绑定本地数据到当前账号
- 清空本地数据后开始

不允许静默自动绑定，避免把旧数据错误并入错误账号。

## 会话与单设备登录

### 登录态机制

- 登录成功后，后端签发 session token。
- 前端把 token 持久化保存到本地。
- 用户以后再次打开 App：
  - 先输入本地进入密码。
  - 再使用本地 token 校验登录态。
  - token 有效则直接进入主界面。
  - token 无效则进入手机号体系。

### 单设备登录规则

- 每个账号同一时间只保留 1 个有效 session。
- 新设备登录成功后，旧设备 session 立即失效。
- 旧设备在后续请求任何受保护接口时，后端返回会话失效或异地登录状态码。
- 前端收到该状态后：
  - 清除本地登录态。
  - 退回计算器页。
  - 提示“账号已在其他设备登录”。

MVP 阶段使用接口返回失效码触发退回，不要求立即引入 websocket 主动通知。

## 后端能力

### 账号接口

- 注册
- 登录
- 获取当前登录用户
- 退出登录
- 修改资料

### 好友接口

- 按手机号搜索用户
- 发送好友申请
- 接受好友申请
- 拒绝好友申请

### 会话接口

- 校验当前登录态
- 注销旧 session / 挤下线

### 聊天数据接口

- 当前账号聊天记录读写
- 当前账号联系人与好友关系读写

## 数据库设计

### users

- id
- phone
- password_hash
- nickname
- avatar_url
- status
- created_at
- updated_at

约束：

- `phone` 唯一
- 密码仅存 `password_hash`，不存明文密码

### user_sessions

- id
- user_id
- device_id
- session_token
- session_version 或 revoked_at
- last_active_at
- created_at

约束：

- 同一 `user_id` 同时只允许 1 条有效会话

### friend_requests

- id
- from_user_id
- to_user_id
- status
- created_at
- handled_at

### friendships

- id
- user_id
- friend_user_id
- created_at

### 聊天数据表

现有聊天表继续使用，但需要把归属统一到账号主键与会话主键体系上，确保聊天、好友、联系人都以 `user_id` 为归属根。

## 前端改造范围

### 新增页面

- 手机号输入页
- 密码登录页
- 注册页
- 补资料页
- 本地数据处理选择页

### 调整现有页面

#### Calculator

- 保留为伪装入口

#### UnlockPinSettings

- 保存进入密码成功后自动返回上一层

#### ChatList

- 去掉顶部“私密聊天服务已连接 / 未连接”状态栏
- 去掉“用户ID添加”
- 去掉“我的二维码”
- 去掉“扫一扫”
- 新增“手机号添加好友”

#### MembershipCenter

- 改为“联系开发者开通会员”
- 去掉上传截图、收款码、付款说明与人工审核提交流程

## 会员策略

- 因为聊天默认上云，会员不再承担“是否允许上云”的开关作用。
- 当前保留“联系开发者开通会员”入口。
- 会员的具体权益后续单独设计，不与本次账号体系改造耦合。

## 安全与风控底线

- 密码必须哈希存储，禁止明文保存。
- 手机号搜索接口必须限流，避免批量枚举。
- 登录接口必须限流，避免撞库。
- session token 必须支持失效与替换。
- 好友搜索结果对外只返回昵称和头像，不返回手机号展示字段。

## 测试重点

### 主流程测试

- 新设备首次注册成功
- 注册后补昵称与头像流程正确
- 首次登录后长期登录态生效
- 关闭并再次打开 App 后无需重新输入手机号登录
- 换设备登录后旧设备被顶下线
- 旧设备退回计算器页并提示异地登录

### 数据测试

- 首次登录时本地旧数据可正确绑定
- 选择清空时旧数据被清除且不串号
- 聊天数据跟账号走，不因设备切换而串号

### 好友测试

- 手机号添加好友只展示昵称与头像
- 未找到用户时统一提示“未找到用户”
- 发送、接受、拒绝好友申请流程正常

### 关联 UI 测试

- 状态栏移除后页面布局正常
- 用户 ID / 二维码入口彻底移除
- 会员页改为联系开发者后流程正确
- 设置密码保存成功后自动返回

## 推荐实施顺序

1. 打通账号表、会话表、注册登录接口。
2. 前端接入“计算器密码后分流”。
3. 完成注册后补资料与首次本地数据处理选择。
4. 替换好友体系为手机号搜索添加。
5. 把聊天数据归属改到账号体系并默认上云。
6. 接入单设备登录与挤下线处理。
7. 收尾调整状态栏、会员页、旧入口、测试号、弹窗与桌面图标。
